## 技术债务详情

### 债务 1：ORM 使用（SQLAlchemy → 原生 SQLite）

#### 📋 债务描述
- **计划要求**：使用 SQLAlchemy 2.0+ 作为 ORM 层（计划书 2.3 节）
- **实际实现**：使用原生 SQLite 直接操作
- **影响范围**：数据库模型层

#### 🚨 原因分析
- **技术限制**：Python 3.13 + SQLAlchemy 2.x 存在兼容性问题
- **错误信息**：`AssertionError: Class <class 'sqlalchemy.sql.elements.SQLCoreOperations'> directly inherits TypingOnly but has additional attributes`
- **解决方案**：使用原生 SQLite 创建表结构

#### 🔧 当前影响
| 影响 | 程度 | 说明 |
|------|------|------|
| 开发效率 | 中 | 需要手写 SQL 语句，失去 ORM 优势 |
| 类型安全 | 低 | 失去编译时类型检查 |
| 数据迁移 | 低 | 表结构简单，迁移风险可控 |
| 维护成本 | 低 | 代码量少，易于理解 |

#### 📝 解决方案评估

##### 方案一：降级 Python 版本
```yaml
方案: 降至 Python 3.12
优点: 完全兼容 SQLAlchemy 2.x
缺点: 影响其他依赖，可能引入新问题
优先级: 中
```

##### 方案二：使用 SQLAlchemy 1.x
```yaml
方案: 使用 SQLAlchemy 1.4
优点: 向后兼容，功能完整
缺点: 未来可能不维护，API 有差异
优先级: 中
```

##### 方案三：维持现状
```yaml
方案: 继续使用原生 SQLite
优点: 稳定，无兼容性问题
缺点: 失去 ORM 的高级特性
优先级: 低（短期）
```

#### 🎯 建议行动
1. **短期**：维持现状，不影响当前开发
2. **中期**：评估项目是否需要部署到 Python 3.12 环境
3. **长期**：如长期使用，建议选择方案一或方案二

### 债务 2：UNIQUE 约束移除（已解决）

#### 📋 债务描述
- **计划要求**：time_series 表有 UNIQUE(metric_id, timestamp) 约束
- **实际实现**：移除 UNIQUE 约束，由业务层处理重复数据
- **状态**：✅ 已解决

#### 🚨 解决原因
- 业务场景需要支持数据重试和批量导入
- 唯一约束限制了业务灵活性
- 重复数据检查在业务层处理更高效

#### 📝 相关变更
- 文件：`backend/init_db_simple.py`
- 移除：`UNIQUE(metric_id, timestamp)` 约束
