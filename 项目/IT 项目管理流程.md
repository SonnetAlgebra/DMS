
## 第一章：核心流程概览（提纲挈领）

|流程|活动|文件|
|---|---|---|
|**设想**|明确项目目标、用户、核心价值；定义核心功能范围与优先级；识别主要风险；指定决策者。|《需求文档》|
|**规划**|将需求拆解为可执行任务；为任务分配排期；负责后续流程的进度跟踪、协调与计划调整。|《计划书》|
|**执行**|按计划完成每个任务；边做边验（每任务完成后自测及同伴验证）；根据验证结果修复缺陷；定期同步进度与阻碍。|代码成果、测试成果（如测试用例、缺陷记录等）|

## 第三章：执行阶段精细化管理（补充规范）

### ### 3.1 核心文件体系（按项目生命周期逻辑排序）

表格

| 文件                   | 角色/受众              | 更新机制             | 核心作用                                                       |
| -------------------- | ------------------ | ---------------- | ---------------------------------------------------------- |
| **项目情况说明.md**        | 项目全貌（全体团队）         | 项目启动时创建，阶段性交付时更新 | **项目背景锚点**：提供项目的整体背景、业务目标和关键约束，确保所有执行决策不偏离初衷。              |
| **CLAUDE.md**        | 执行指南（开发AI每次自动静默读取） | 架构变更或规范更新时手动维护   | **技术规范基石**：固化业务红线、代码规范、错误处理策略，作为开发AI的本地执行准则，确保代码质量一致性。     |
| **PROJECT_STATE.md** | 进度追踪（供项目经理与老板查阅）   | 任务节点完成时开发AI自动更新  | **进度同步枢纽**：记录当前任务状态、待办事宜及下一步计划，充当团队协作的"记忆锚点"和断点续传依据。       |
| **Git Commit记录**     | 技术档案（供技术负责人审查）     | 功能闭环时开发AI自动提交    | **技术演化记录**：精准记录代码差异（diff）与修改逻辑，取代单独的日志文本，为技术审查和问题追溯提供原始依据。 |

### 3.2 执行工作流规约

**启动与同步流程：**

- 开发AI启动会话时，自动按序读取：项目情况说明.md → CLAUDE.md → PROJECT_STATE.md
- 向老板汇报："已确认项目背景、技术规范和最新进度，当前任务为[X]，是否开始？"

**任务执行标准：**

- 严格遵循CLAUDE.md规范编写代码，确保与项目情况说明.md中的业务目标一致
- 运行本地自测（如pytest或验证API）
- 自测通过后提交Git Commit（详细注明修改详情），并更新PROJECT_STATE.md

**异常处理机制：**

- 遇到技术阻塞时，自动尝试修复并重测
- 连续3次尝试失败时，触发**强制挂起**
- 输出"【请求协助】" + 错误堆栈 + 涉及代码变更，通过老板网关传递给技术负责人AI

### 3.3 开发AI执行戒律

- **背景导向**：任何技术决策必须回溯到项目情况说明.md中的业务目标
- **规范优先**：严格按CLAUDE.md定义的数据库模型与技术栈执行，禁止私自引入未授权依赖
- **进度透明**：PROJECT_STATE.md必须实时反映真实进度，禁止滞后更新
- **质量闭环**：不测不交，任何代码提交前必须经过本地验证
- **追溯完整**：技术细节和参数变更必须体现在Git Commit Message中
- **熔断机制**：同一错误连续修复3次未解决，立即停止并输出完整上下文
